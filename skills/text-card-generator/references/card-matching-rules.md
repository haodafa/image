# 卡片匹配规则

## 适合做成卡片的内容

当文本中出现以下结构化信息时，应考虑生成卡片图：

- **并列要点**：3 条以上带标题+描述的条目（特性列表、核心能力、优势罗列）→ `list`
- **对比结构**：两组对立的条目，如"之前/之后"、"优点/缺点"、"适合/不适合" → `comparison` 或 `fit`
- **流程步骤**：有序的操作步骤、循环流程、判断分支 → `flow`
- **命令行演示**：shell 命令序列、终端输出、代码示例需要增强表现 → `terminal`
- **核心结论**：单条重要观点、金句、slogan → `quote`
- **数据指标**：数字+标签的组合（性能指标、统计数据、版本号） → `stats`
- **架构层级**：组件分层、系统拓扑、模块关系 → `architecture`

## 不适合做成卡片的内容

- 叙述性段落（背景介绍、过渡段、个人感受）
- 已有图片的段落（除非用户明确要求替换）
- 单行结论（太短，做成图反而浪费空间）
- 代码块需要读者复制的场景（图片里的代码不可复制）
- 引用的原文段落（保持原始格式更好）

## 类型选择优先级

同一段落可能匹配多种类型时，按以下优先级选择：

1. 有明确正反两组 → `comparison` 或 `fit`
2. 有明确顺序/因果 → `flow`
3. 有代码/命令 → `terminal`
4. 有数字指标 → `stats`
5. 是并列条目 → `list`
6. 是单条结论 → `quote`

## 合并与拆分

- 相邻的同类型内容应合并为一张卡片（如连续的两个列表）
- 单张卡片条目超过 8 条时考虑拆分为多张
- `comparison` 两侧条目数应尽量对等（差距超过 2 条时调整内容）
